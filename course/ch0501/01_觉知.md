## 1. 为什么要学习 MCP 开发

通过前期对 MCP 应用的了解与使用，我们已经认识并体验了大量的现有应用。然而，在实际应用中，经常会发现市场上现有的 MCP 应用并不能完全满足自身高度个性化的需求。为了满足自身高度个性化的定制需求以及对 MCP 应用的深度控制，我们需要从单纯的"使用者"转向"创造者"，开始开发专属于自己的 MCP 应用。

通过亲手开发，我们能够更深入地了解 MCP 应用的内部运作机制，能让我们更精准地评判一个 MCP 应用的质量和优劣。此外，深入参与开发也使我们能够更全面地认识到 MCP 协议本身的局限性。只有理解了这些局限所在，我们才能更好地把握 MCP 的最佳应用场景，从而更有效地利用这一技术，甚至思考 MCP 的未来发展方向，并参与到这个不断完善的系统建设中来。

## 2. MCP 的三大核心组件

掌握 MCP 的核心知识是成功开发 MCP 应用的基础。

在开发者的视角下，MCP 本质上是一套标准协议，它定义了客户端（Client）与服务器端（Server）如何进行交互和协作。当前，由于客户端大多已实现 MCP 的相关功能，因此开发者通常只需要将工作重点放在构建 MCP Server 上，以实现具体的功能和服务。

为了让 Server 能有效地提供服务并与客户端沟通，我们必须理解构成 MCP Server 能力基石的三大核心组件：

### 第一个核心组件：工具（Tool）

工具的作用是帮助大型语言模型（LLM）执行那些在对话中难以实现的操作或特定的功能。例如，查询天气、发送邮件、或者进行数学计算等都需要通过工具来实现。

**核心特点**：
- 可调用的函数
- 决定了 MCP 可以做什么
- 用于执行操作

### 第二个核心组件：资源（Resource）

资源与工具的作用有所不同：工具用于操作，而资源则主要用于提供 Server 端的特定数据。

**核心特点**：
- 资源通常是**只读的**，客户端可以查看但不能对其进行更新或操作
- 用于提供上下文信息（例如代办事项、用户资料、天气数据、菜单列表或文件内容等）
- 每个资源都通过一个 URI（类似于 URL）进行唯一标识
- 支持**订阅机制**，客户端可以接收到关于资源更新的通知
- 决定了 MCP 可以有什么

### 第三个核心组件：提示模板（Prompt Template）

这个组件容易与普通的提示词（prompt）相混淆。提示模板本质上是一种模版，由 Server 端预先配置好并提供给 Client 端使用。

**核心价值**：
- 规范客户端（即 LLM）如何调用工具
- 通过提供预先写好的提示模板（例如代码审查模板或数据分析模板），Server 可以确保 LLM 在特定情境下以更可靠、更稳定的方式去调用相应的工具
- 避免模型随意使用各种方式调用工具而导致的不确定性
- 决定了 MCP 的复用能力

---

## 总结

总之，MCP 的开发就是围绕着如何有效地组织和设计这些核心组件展开：

- **工具**负责执行操作
- **资源**负责提供只读数据
- **提示模板**负责指导客户端如何稳定地利用这些能力
