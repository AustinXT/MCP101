## 综合知识点（来自优秀作业）

- 规格文档驱动开发（SPEC）
  - 以规范先行，明确工具接口、参数、约束与预期行为，再进入实现与评估。
  - 参考资料：[`github/spec-kit`](https://github.com/github/spec-kit)
- 三角色流程与职责边界
  - 架构师：需求澄清、API 调研与选型、交互模式（如 Stdio）与工具参数设计、删繁就简。
  - 开发者：严格对齐规范实现能力（如“生成图片”“批量生成图片”），首要保证可用性与可测试性。
  - 评估师：基于规范验收、发现可用性问题（如返回 URL 需下载）并推动迭代闭环。
- 端到端演练要点（以 AI 画图 MCP 为例）
  - 外部 API 调研→工具定义→实现→验证→迭代，形成小步快跑的闭环。
  - 交互通道优先采用本地 `Stdio` 降低门槛；参数设计去除非核心选项（如“水印”）。
- 工程与环境
  - 使用 `uv` 等虚拟环境管理与本地运行 MCP Server，降低首次配置成本。
  - 持续迭代修复报错与体验问题（如下载逻辑、本地保存等）。
- 实战经验与坑
  - API Key 与权限开通需前置检查；对复杂 API 文档要做“人类可读”的二次提炼。
  - 模型与通道选型直接影响效果与效率，必要时替换更“稳/快/准”的模型。

## 实践建议

- 从规范出发，先定能力边界与输入输出，再写代码与用例。
- 将“评估师”环节制度化：每次提交都附带对照规范的验收点与问题清单。
- 优先保障端到端“可走通”，随后再做参数细化与体验优化。
